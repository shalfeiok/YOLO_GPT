# Решение по `tk_ui` (CustomTkinter)

Дата: 2026-02-20
Статус: **Accepted**

## Короткий ответ
`tk_ui` **не нужен в production-пути приложения**.

Проект стандартизирован на **PySide6 (Qt)** как единственный GUI-стек для основного запуска (`main.py`).

## Почему
- Два GUI-стека (Qt + Tk) увеличивают стоимость поддержки и регрессы.
- Headless/CI окружения стабильнее без Tk-зависимостей в runtime.
- В коде `app/features/*/ui.py` уже используются безопасные stubs, которые явно указывают на перенос legacy UI в `examples/tk_ui/`.

## Что оставляем
- `examples/tk_ui/` оставляем как **архив/справочник** legacy реализации.
- В production-коде (`app/*`) Tk-реализации не используются.

## Политика дальше
1. Новые UI-фичи — только в PySide6.
2. Любые ссылки на Tk в `app/*` должны оставаться только в виде stubs/сообщений о переносе в `examples/tk_ui`.
3. Если понадобится отдельный legacy launcher для Tk, он должен жить вне production entrypoint и быть явно опциональным.
